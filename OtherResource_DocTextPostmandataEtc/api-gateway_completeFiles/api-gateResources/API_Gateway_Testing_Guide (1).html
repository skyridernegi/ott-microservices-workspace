🔑 JWT Token — Generate and Use

What is JWT? (Interview answer)

JWT = JSON Web Token. Three parts separated by dots:

eyJhbGciOiJIUzI1NiJ9  .  eyJ1c2VybmFtZSI6ImFkbWluIn0  .  SflKxwRJSMeKKF2QT4fwpM
     HEADER                      PAYLOAD                         SIGNATURE
   (algorithm)              (username, role, expiry)          (tamper proof)

Stateless — user data lives IN the token. No server session needed across microservices.

AGenerate JWT Token — POST /auth/tokenPUBLIC — no token needed
POST http://localhost:8080/auth/token

In Postman → Body tab → raw → select JSON from dropdown → paste:

{
  "username": "john.doe",
  "role": "USER"
}
✅ 200 OK Response: { "token": "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJqb2huLmRvZSIsInJvbGUiOiJVU0VSIn0.abc123xyz...", "username": "john.doe", "role": "USER", "expiresIn": "1 hour" }
⚠️ Copy the full token value (starts with eyJ...) — you need it for every protected API call.

You can also generate an ADMIN token:

{
  "username": "admin.user",
  "role": "ADMIN"
}

⚡ Rate Limiter Testing — Windows 10

How Rate Limiter Works (Interview Answer)

Uses Token Bucket algorithm with Redis:

  • Each user gets a bucket with 10 tokens per second stored in Redis
  • Each request consumes 1 token from the bucket
  • Bucket refills at 10 tokens/second continuously
  • When bucket is empty → Gateway returns 429 Too Many Requests
  • Key = username from JWT token (each user has their own bucket)
Option 1Postman Collection Runner (Easiest)
1. In Postman left sidebar → click Collections
2. Hover over OTT Platform collection → click ▶ Run button
3. Select only GET /catalog/plans request
4. Set Iterations = 25
5. Set Delay = 0 ms (important — 0 delay fires rapidly)
6. Click Run OTT Platform
Expected results: Request 1 → 200 OK ✅ Request 2 → 200 OK ✅ ... Request 11 → 429 Too Many Requests ❌ ← rate limiter kicked in! Request 12 → 429 Too Many Requests ❌ ... Request 25 → 429 Too Many Requests ❌

Open PowerShell (Windows key → type powershell → Enter)

Replace YOUR_TOKEN_HERE with your actual JWT token → paste entire script → press Enter:

🎯 Interview Q&A — Click to expand answers

Q1: What is API Gateway and why do we need it? ▼
API Gateway is the single entry point for all client requests in a microservices architecture. Without it, the client must know the address of every microservice and handle cross-cutting concerns like authentication separately for each one. Gateway centralizes routing, JWT authentication, rate limiting, and logging in one place. In our OTT platform, the ESPN website calls port 8080 only — Gateway routes to catalog-service or subscription-service based on the URL path.
Q2: How does JWT authentication work in your gateway? ▼
Every request hits JwtAuthFilter first (it's a GlobalFilter with Order=1). It checks the Authorization header for "Bearer token". If missing or not starting with "Bearer" — returns 401 immediately. If present — extracts the token, validates it using HMAC-SHA256 signature via JwtUtil. If invalid or expired — returns 401. If valid — extracts username and role, adds them as X-Username and X-Role headers, and forwards the modified request to the downstream microservice. The microservice never handles authentication — Gateway does it all.
Q3: What is rate limiting and how did you implement it? ▼
Rate limiting controls how many requests a user can make in a given time window to prevent abuse and protect downstream services. I used Spring Cloud Gateway's RequestRateLimiter filter with Redis as the backend store. It uses the Token Bucket algorithm — each user gets a bucket of 10 tokens per second stored in Redis. Each request consumes one token. When the bucket is empty, Gateway returns 429 Too Many Requests immediately without forwarding to the microservice. The key resolver uses the X-Username header (set by JwtAuthFilter) so each user has their own independent bucket.
Q4: What is the difference between GlobalFilter and GatewayFilter? ▼
GlobalFilter applies to ALL routes automatically — used for JWT authentication and logging which every request needs regardless of which service it's going to. GatewayFilter applies to specific routes only and is configured per route in application.yml — used for rate limiting which can have different limits per route. In our project: JwtAuthFilter and LoggingFilter implement GlobalFilter, while RequestRateLimiter is a GatewayFilter defined in each route in application.yml.
Q5: Why does Gateway use WebFlux instead of Spring MVC? ▼
Gateway handles thousands of simultaneous requests. Traditional Spring MVC is blocking — one thread per request, thread waits for the response before handling the next request. WebFlux is non-blocking and reactive — one thread can handle many requests concurrently using event-driven programming. This makes Gateway extremely efficient without consuming thousands of threads. This is also why we use Spring Data Redis Reactive — mixing blocking Redis with reactive Gateway causes startup errors.
Q6: What is JWT and why is it better than session-based auth for microservices? ▼
JWT (JSON Web Token) is a stateless token containing user data (username, role, expiry) signed with a secret key. Traditional sessions store user data on the server — in a microservices architecture with 10+ services, there's no single server to store sessions, so session-based auth doesn't scale. JWT stores the data in the token itself — any service can validate it by checking the signature without calling a central server. This is called stateless authentication and is the industry standard for microservices.
Q7: How do your microservices communicate with each other? ▼
Subscription-service calls catalog-service synchronously via REST using RestTemplate (or Feign Client). When a user subscribes, subscription-service calls GET /catalog/plans/{planId} on catalog-service to validate the plan and get the price. This is synchronous REST communication. In more complex scenarios like payment failures and subscription expiry, we would use asynchronous event-driven communication via Kafka or AWS Kinesis — the next phase of our architecture.
Q8: What happens if catalog-service is down when subscription-service calls it? ▼
Currently, if catalog-service is down, our PlanServiceException is thrown and GlobalExceptionHandler returns a 502 Bad Gateway error to the client. In a production system, we would add a Circuit Breaker pattern using Resilience4j — it monitors failures and "opens the circuit" after a threshold, returning a fallback response instead of waiting for timeout. This prevents cascade failures where one service being down brings down the entire system. This is called the Circuit Breaker pattern and is a key microservices resilience pattern.

How to introduce your project in interview

"I built an OTT subscription platform similar to Disney+/ESPN. It has three components — an API Gateway on port 8080 built with Spring Cloud Gateway that handles JWT authentication and rate limiting using Redis, a Catalog Service on port 8081 that manages OTT plans in MySQL, and a Subscription Service on port 8082 that handles user subscriptions. The subscription service calls the catalog service synchronously via REST to validate plans before subscribing. Each service has its own database following the database-per-service pattern. I used centralized exception handling with @RestControllerAdvice, DTOs for clean API contracts, and soft deletes for data integrity."
