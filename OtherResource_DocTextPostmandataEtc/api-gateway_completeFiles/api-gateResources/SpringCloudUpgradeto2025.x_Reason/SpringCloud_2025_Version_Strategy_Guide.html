🚀 Spring Cloud 2025.0.0 — Version Strategy Guide

Why upgrade · Official compatibility · Minimal changes · Future-proof for Eureka, Circuit Breaker, Kafka

Why your thinking is correct — Senior Architect perspective

❌ Wrong approach (what we had)
  • catalog-service: Boot 3.5.14
  • subscription-service: Boot 3.5.14
  • api-gateway: Boot 3.4.5 ← downgraded!
  • Spring Cloud: 2024.0.1 (mixed)
  • Problems: version mismatch between services, Eureka will fail, future upgrades break one service at a time
✅ Correct approach (what we're doing)
  • catalog-service: Boot 3.5.14
  • subscription-service: Boot 3.5.14
  • api-gateway: Boot 3.5.14 ← back to same!
  • Spring Cloud: 2025.0.0 everywhere
  • Benefits: consistent stack, all future features work across all services equally

Spring Cloud 2025.0.0 (codename "Northfields") released May 2025 — officially compatible with Spring Boot 3.5.x. This is the currently active supported version.

catalog-service — pom.xml changes

PURPOSE of adding Spring Cloud to catalog-service:
catalog-service doesn't USE Spring Cloud features right now. But when we add Eureka next, catalog-service needs to register itself. Adding the BOM now means zero changes to pom.xml when Eureka comes — just add the dependency, version is already managed.
1
pom.xml — Add Spring Cloud BOM (2 changes only)
pom.xml MODIFY catalog-service/pom.xml

CHANGE 1 — Add version property after <java.version>

<properties>
    <java.version>21</java.version>
    <spring-cloud.version>2025.0.0</spring-cloud.version>
</properties>

CHANGE 2 — Add Spring Cloud BOM after </properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>${spring-cloud.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
That's it for catalog-service. No other changes. Maven Update Project → done.

api-gateway — pom.xml changes

What happened before: Gateway had Spring Boot 3.5.14 + Spring Cloud 2024.0.1 → incompatible → we downgraded Boot to 3.4.5 as a workaround.

Correct fix now: Keep Boot 3.5.14 and upgrade Cloud to 2025.0.0 — that's the compatible pair.
1
pom.xml — Upgrade Boot back to 3.5.14 + Cloud to 2025.0.0
pom.xml MODIFY api-gateway/pom.xml

CHANGE 1 — Upgrade Spring Boot version in parent block

<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.4.5</version>  <!-- OLD — remove this -->
    <version>3.5.14</version> <!-- NEW — same as other services -->
    <relativePath/>
</parent>

CHANGE 2 — Update Spring Cloud version property

<properties>
    <java.version>21</java.version>
    <spring-cloud.version>2024.0.1</spring-cloud.version>  <!-- OLD -->
    <spring-cloud.version>2025.0.0</spring-cloud.version>  <!-- NEW -->
</properties>
After these 2 changes:
Right-click api-gateway → Maven → Update Project → OK
Then restart api-gateway → check console for startup errors.
Spring Cloud 2025.0.0 deprecated some Gateway module names. If you see deprecation warnings they are warnings only — not errors — everything still works.
2
application.yml — One property to add for Gateway
application.yml MODIFY api-gateway/src/main/resources/application.yml

Spring Cloud 2025.0.0 changed a default behaviour — X-Forwarded headers are now disabled by default for security. Add this to maintain correct request forwarding to microservices:

spring:
  cloud:
    gateway:
      server:
        webflux:
          trusted-proxies: ".*"  # trust all proxies — for local dev only
          # In production replace .* with your actual proxy IP range
          # e.g. "10\\.0\\.0\\..*"
This setting is new in Spring Cloud 2025.0.0 as a security improvement. For local development ".*" (trust all) is fine. In production you would restrict to your actual load balancer IP.

Verify all changes are correct — step by step

1
Apply all pom.xml changes in STS
  1. Open catalog-service/pom.xml → add spring-cloud.version property + BOM → save → right-click catalog-service → Maven → Update Project → OK
  2. Open subscription-service/pom.xml → add spring-cloud.version + BOM + Feign dependency → save → Maven Update Project
  3. Open api-gateway/pom.xml → change Boot to 3.5.14 + change Cloud to 2025.0.0 → save → Maven Update Project
  4. Open api-gateway/application.yml → add trusted-proxies setting
2
Start services and verify console output

Start in order: catalog → subscription → gateway

catalog-service console — look for:

Started CatalogServiceApplication on port 8081
  (no Spring Cloud errors)

subscription-service console — look for:

Started SubscriptionServiceApplication on port 8082
  (Feign clients registered — if you added Feign files)

api-gateway console — look for:

Started ApiGatewayApplication on port 8080
  (no compatibility errors — the one you had before is gone now)
3
Quick Postman verification

POST http://localhost:8080/auth/token → get token

GET http://localhost:8080/catalog/plans → 200 OK with plans list

POST http://localhost:8080/subscriptions → 201 Created (Feign calls catalog)

If all three work → upgrade complete. All services on same consistent version stack.
