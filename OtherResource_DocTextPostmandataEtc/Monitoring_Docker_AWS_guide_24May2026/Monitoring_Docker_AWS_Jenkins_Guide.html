What is monitoring in microservices?
In production, you need to know: Is my service alive? How much memory is it using? How many requests are failing? How slow are my DB queries?

Spring Boot Actuator exposes built-in HTTP endpoints that answer all these questions. No extra code needed — just add the dependency and configure.

Step 1 — Add Actuator dependency to ALL 3 services

1Add to pom.xml — catalog-service, subscription-service, api-gateway
pom.xmlALL 3 SERVICESAdd inside <dependencies>

WHY: Actuator adds /actuator/* endpoints automatically. Health check, metrics, info, environment — all free with this one dependency.

<!-- Spring Boot Actuator — monitoring endpoints -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
After adding to all 3 pom.xml files → right-click each project → Maven → Update Project → OK

Step 2 — Configure Actuator in each service's properties

2Acatalog-service — application.properties
application.propertiesMODIFYcatalog-service/src/main/resources

WHY these settings: By default Actuator only exposes /health. We expose more endpoints so we can see metrics, info, and environment details during development.

# ── Spring Boot Actuator — Monitoring ─────────────────────────
# Expose all actuator endpoints (restrict in production)
# health = is service alive?
# info = app version, name
# metrics = memory, CPU, request counts
# env = environment properties
management.endpoints.web.exposure.include=health,info,metrics,env,loggers
management.endpoint.health.show-details=always
management.info.env.enabled=true

# App info shown at /actuator/info
info.app.name=Catalog Service
info.app.version=1.0.0
info.app.description=OTT Platform - Manages OTT subscription plans

What is Docker and why do microservices need it?
Docker packages your Java app + all its dependencies into a container — a self-contained box that runs identically on your Windows laptop, your team member's Mac, and the AWS server. No "it works on my machine" problem.

CONNECTS TO: Dockerfile defines the image → docker-compose.yml runs all services together → AWS ECR stores images → EC2 runs containers.

Step 4 — Create docker-compose.yml (runs everything together)

4docker-compose.yml — ONE file to start the entire platform
docker-compose.ymlNEW FILECreate in a new folder: ott-platform/ (parent of all 3 service folders)

WHY: Instead of starting Redis, MySQL, and 3 Spring Boot apps manually, one command starts everything. This is what DevOps teams use locally.

version: '3.8'

services:

  # ── MySQL Database ─────────────────────────────
  mysql:
    image: mysql:8.0
    container_name: ott-mysql
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: catalog_db
    ports:
      - "3306:3306"
    volumes:
      - mysql_data:/var/lib/mysql   # persist data between restarts
      - ./database_schema.sql:/docker-entrypoint-initdb.d/schema.sql
    networks:
      - ott-network
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ── Redis ──────────────────────────────────────
  redis:
    image: redis:7-alpine
    container_name: ott-redis
    ports:
      - "6379:6379"
    networks:
      - ott-network

  # ── Catalog Service ────────────────────────────
  catalog-service:
    build:
      context: ./catalog-service     # folder containing Dockerfile
    container_name: ott-catalog
    ports:
      - "8081:8081"
    environment:
      # Override application.properties for Docker environment
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/catalog_db?useSSL=false&allowPublicKeyRetrieval=true
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: root
    depends_on:
      mysql:
        condition: service_healthy   # wait for MySQL to be ready
    networks:
      - ott-network

  # ── Subscription Service ───────────────────────
  subscription-service:
    build:
      context: ./subscription-service
    container_name: ott-subscription
    ports:
      - "8082:8082"
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/subscription_db?useSSL=false&allowPublicKeyRetrieval=true
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: root
      CATALOG_SERVICE_URL: http://catalog-service:8081  # use container name not localhost
    depends_on:
      mysql:
        condition: service_healthy
      catalog-service:
        condition: service_started
    networks:
      - ott-network

  # ── API Gateway ────────────────────────────────
  api-gateway:
    build:
      context: ./api-gateway
    container_name: ott-gateway
    ports:
      - "8080:8080"
    environment:
      SPRING_DATA_REDIS_HOST: redis  # use container name not localhost
    depends_on:
      - redis
      - catalog-service
      - subscription-service

Manual AWS deployment — what we're doing:
Push Docker images to AWS ECR (image registry) → Launch EC2 instance (virtual server) → Pull images on EC2 → Run with docker-compose on EC2.

This is the simplest deployment path — good to understand before adding Jenkins automation.

1AWS account + CLI setup on Windows 10

2. Download AWS CLI for Windows: https://aws.amazon.com/cli → download MSI installer → install

3. Configure AWS CLI — open Command Prompt:

Step 2 — Push Docker images to AWS ECR

2Create ECR repositories and push images

WHY ECR: ECR (Elastic Container Registry) is AWS's Docker image storage. EC2 pulls images from here. Like Docker Hub but private and integrated with AWS.

Open Command Prompt and run these commands one by one:

REM Step 1: Login to ECR (replace ACCOUNT_ID and REGION)
aws ecr get-login-password --region ap-south-1 | docker login --username AWS --password-stdin ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com

REM Step 2: Create repositories in ECR
aws ecr create-repository --repository-name ott/catalog-service --region ap-south-1
aws ecr create-repository --repository-name ott/subscription-service --region ap-south-1
aws ecr create-repository --repository-name ott/api-gateway --region ap-south-1

REM Step 3: Build images locally
docker build -t ott/catalog-service ./catalog-service
docker build -t ott/subscription-service ./subscription-service
docker build -t ott/api-gateway ./api-gateway

REM Step 4: Tag images for ECR
docker tag ott/catalog-service:latest ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/catalog-service:latest
docker tag ott/subscription-service:latest ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/subscription-service:latest
docker tag ott/api-gateway:latest ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/api-gateway:latest

REM Step 5: Push to ECR
docker push ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/catalog-service:latest
docker push ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/subscription-service:latest
docker push ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com/ott/api-gateway:latest

What is Jenkins CI/CD?
Every time you push code to GitHub, Jenkins automatically: builds your code → runs tests → builds Docker image → pushes to ECR → deploys to EC2. No manual steps needed.

What is CloudFormation?
Instead of manually clicking in AWS Console to create EC2, RDS, security groups — you write a YAML file describing your infrastructure. CloudFormation reads it and creates everything automatically. Infrastructure as Code.

CloudFormation Template — complete AWS infrastructure

1cloudformation.yml — creates entire AWS infrastructure
cloudformation.ymlNEW FILEott-platform/aws/
AWSTemplateFormatVersion: '2010-09-09'
Description: OTT Platform — Complete AWS Infrastructure

Parameters:
  KeyPairName:
    Type: String
    Description: EC2 Key Pair name for SSH access
    Default: ott-key

Resources:

  # ── VPC — Virtual Private Cloud ───────────────────────────────
  # WHY: Isolated network for our services. Like a private office network.
  OttVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16
      Tags:
        - Key: Name
          Value: ott-platform-vpc

  # ── Public Subnet ─────────────────────────────────────────────
  OttPublicSubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref OttVPC
      CidrBlock: 10.0.1.0/24
      MapPublicIpOnLaunch: true
      AvailabilityZone: !Select [0, !GetAZs '']

  # ── Internet Gateway — allows internet access ─────────────────
  OttInternetGateway:
    Type: AWS::EC2::InternetGateway

  AttachGateway:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties:
      VpcId: !Ref OttVPC
      InternetGatewayId: !Ref OttInternetGateway

  # ── Security Group — firewall rules ───────────────────────────
  OttSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: OTT Platform security group
      VpcId: !Ref OttVPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 22
          ToPort: 22
          CidrIp: 0.0.0.0/0    # SSH — restrict to your IP in production
        - IpProtocol: tcp
          FromPort: 8080
          ToPort: 8080
          CidrIp: 0.0.0.0/0    # API Gateway
        - IpProtocol: tcp
          FromPort: 8081
          ToPort: 8082
          CidrIp: 0.0.0.0/0    # Microservices

  # ── EC2 Instance — runs our Docker containers ──────────────────
  OttEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: t2.micro        # free tier
      ImageId: ami-0c55b159cbfafe1f0  # Amazon Linux 2 (update for your region)
      KeyName: !Ref KeyPairName
      SubnetId: !Ref OttPublicSubnet
      SecurityGroupIds:
        - !Ref OttSecurityGroup
      UserData:
        Fn::Base64: |
          #!/bin/bash
          yum update -y
          yum install docker -y
          service docker start
          usermod -a -G docker ec2-user
          # Install docker-compose
          curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
          chmod +x /usr/local/bin/docker-compose
      Tags:
        - Key: Name
          Value: ott-platform-server

  # ── RDS MySQL — managed database ──────────────────────────────
  # WHY RDS instead of MySQL in Docker: RDS handles backups,
  # failover, patches automatically. Production standard.
  OttDBSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      DBSubnetGroupDescription: OTT DB subnet group
      SubnetIds:
        - !Ref OttPublicSubnet

  OttRDSInstance:
    Type: AWS::RDS::DBInstance
    Properties:
      DBInstanceIdentifier: ott-mysql
      DBInstanceClass: db.t3.micro   # free tier
      Engine: mysql
      MasterUsername: root
      MasterUserPassword: OttPlatform2024
      AllocatedStorage: 20
      DBSubnetGroupName: !Ref OttDBSubnetGroup

Outputs:
  EC2PublicIP:
    Description: Public IP of EC2 instance
    Value: !GetAtt OttEC2Instance.PublicIp

  RDSEndpoint:
    Description: RDS MySQL endpoint
    Value: !GetAtt OttRDSInstance.Endpoint.Address

Jenkins Pipeline — Jenkinsfile

2Jenkinsfile — complete CI/CD pipeline
JenkinsfileNEW FILEott-platform/ (root — commit to GitHub)

WHY: Every time you push to GitHub, Jenkins reads this file and runs each stage automatically.

pipeline {
    agent any

    environment {
        AWS_REGION = 'ap-south-1'
        ECR_REGISTRY = 'ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com'
        IMAGE_TAG = "${BUILD_NUMBER}"  // uses Jenkins build number as tag
    }

    stages {

        // Stage 1: Get code from GitHub
        stage('Checkout') {
            steps {
                git branch: 'main',
                    url: 'https://github.com/YOUR_GITHUB/ott-platform.git'
                echo 'Code checked out successfully'
            }
        }

        // Stage 2: Build all services
        stage('Build') {
            steps {
                dir('catalog-service') {
                    bat 'mvn clean package -DskipTests'
                }
                dir('subscription-service') {
                    bat 'mvn clean package -DskipTests'
                }
                dir('api-gateway') {
                    bat 'mvn clean package -DskipTests'
                }
            }
        }

        // Stage 3: Run unit tests
        stage('Test') {
            steps {
                dir('catalog-service') {
                    bat 'mvn test'
                }
                dir('subscription-service') {
                    bat 'mvn test'
                }
            }
        }

        // Stage 4: Build Docker images
        stage('Docker Build') {
            steps {
                bat "docker build -t ${ECR_REGISTRY}/ott/catalog-service:${IMAGE_TAG} ./catalog-service"
                bat "docker build -t ${ECR_REGISTRY}/ott/subscription-service:${IMAGE_TAG} ./subscription-service"
                bat "docker build -t ${ECR_REGISTRY}/ott/api-gateway:${IMAGE_TAG} ./api-gateway"
            }
        }

        // Stage 5: Push images to AWS ECR
        stage('Push to ECR') {
            steps {
                withAWS(credentials: 'aws-credentials', region: AWS_REGION) {
                    bat "aws ecr get-login-password --region ${AWS_REGION} | docker login --username AWS --password-stdin ${ECR_REGISTRY}"
                    bat "docker push ${ECR_REGISTRY}/ott/catalog-service:${IMAGE_TAG}"
                    bat "docker push ${ECR_REGISTRY}/ott/subscription-service:${IMAGE_TAG}"
                    bat "docker push ${ECR_REGISTRY}/ott/api-gateway:${IMAGE_TAG}"
                }
            }
        }

        // Stage 6: Deploy to EC2
        stage('Deploy to EC2') {
            steps {
                withAWS(credentials: 'aws-credentials', region: AWS_REGION) {
                    // SSH to EC2 and pull + restart containers
                    bat """
                        ssh -i ott-key.pem -o StrictHostKeyChecking=no ec2-user@EC2_PUBLIC_IP ^
                        "docker-compose pull && docker-compose up -d"
                    """
                }
            }
        }
    }

    post {
        success {
            echo 'Pipeline completed successfully! OTT Platform deployed.'
        }
        failure {
            echo 'Pipeline failed! Check logs above for errors.'
        }
    }
}

Top interview questions — Monitoring, Docker, AWS, Jenkins

Q: What is Spring Boot Actuator and how did you use it? ▼
Spring Boot Actuator provides production-ready monitoring endpoints out of the box. I added it to all three services in our OTT platform. The /actuator/health endpoint is used by AWS load balancers to check if a service is alive. /actuator/metrics provides JVM memory, CPU, and HTTP request statistics. For the API Gateway, I also exposed /actuator/gateway/routes which shows all configured routing rules. I added /actuator to the public paths in JwtAuthFilter so monitoring tools can call health checks without JWT tokens.
Q: What is Docker and why do microservices need it? ▼
Docker packages an application and all its dependencies into a container — a portable, isolated unit that runs identically everywhere. Without Docker, each microservice needs specific Java version, OS configurations, and library installations on every server. With Docker, you build once and run anywhere. In our OTT platform, I wrote a Dockerfile for each service using the eclipse-temurin:21 base image. Docker Compose coordinates all services, MySQL, and Redis starting together in the correct order with proper networking between them.
Q: What is CloudFormation and how is it different from manual AWS setup? ▼
CloudFormation is AWS's Infrastructure as Code service. Instead of manually clicking through the AWS console to create EC2 instances, VPCs, security groups, and RDS databases, you write a YAML template describing the desired infrastructure. CloudFormation reads it and creates everything automatically. The key benefits are repeatability (same template creates identical environments), version control (infrastructure changes are tracked in Git), and rollback (if deployment fails, CloudFormation rolls back automatically). I wrote a template that creates VPC, subnets, security groups, EC2 instance, and RDS MySQL for our OTT platform with a single deploy command.
Q: What is Jenkins and what is CI/CD? ▼
CI/CD stands for Continuous Integration and Continuous Deployment. Jenkins is an automation server that implements CI/CD. In our OTT platform, the Jenkins pipeline has 6 stages: Checkout code from GitHub, Build all Maven projects, Run unit tests, Build Docker images, Push images to AWS ECR, Deploy to EC2 by pulling new images and restarting containers. This means every code push is automatically built, tested, and deployed without any manual steps. The Jenkinsfile lives in the root of the repository — Jenkins reads it and executes the pipeline.
Q: What is the difference between ECR and Docker Hub? ▼
Both are Docker image registries — storage for Docker images. Docker Hub is public and general purpose. AWS ECR (Elastic Container Registry) is private and deeply integrated with AWS services. ECR images can be pulled by EC2, ECS, and EKS using IAM roles without separate authentication credentials. ECR is in the same AWS region as your compute resources which means faster image pull times and no data transfer costs. For production AWS deployments, ECR is the standard choice over Docker Hub.
Q: How do microservices communicate in Docker Compose vs locally? ▼
Locally, services use localhost with different ports — subscription-service calls catalog-service at http://localhost:8081. In Docker Compose, each service runs in its own container and localhost refers to that container only, not others. So you use the service name defined in docker-compose.yml as the hostname — subscription-service calls catalog-service at http://catalog-service:8081. Docker's internal DNS resolves container names to IPs automatically. This is why I have CATALOG_SERVICE_URL as an environment variable that gets overridden in docker-compose.yml.
