About OAuth2 — do you need it?

OAuth2 vs JWT — honest answer

What OAuth2 is: A protocol for authorization. "Sign in with Google" is OAuth2. When you click Google/Facebook login, that website is using OAuth2 to get your identity from Google without handling your password.

What you already have: JWT-based authentication. User logs in with username/password → your user-service generates a JWT → client sends JWT in every request → API Gateway validates it.

For your project: JWT is sufficient and more commonly asked in interviews for backend roles. OAuth2 is needed if you want "Sign in with Google/Apple" on the ESPN website. For now add it as a talking point: "We use JWT for internal auth; OAuth2 could be added for social login integration."

Interview answer: "Our platform uses JWT-based stateless authentication via a dedicated user-service. The API Gateway validates tokens on every request. We could extend this with OAuth2/OIDC for social login integration, but for an OTT subscription platform, JWT is the industry standard."

Service Discovery with Eureka — concept + implementation

What is Service Discovery and WHY do we need it?

Problem without it: In your current code, subscription-service calls catalog-service at hardcoded http://localhost:8081. On AWS, services run on different servers with different IPs. When you scale to 3 catalog-service instances, which IP does subscription-service call?

Solution — Eureka: Every service registers itself with Eureka on startup ("Hi, I'm catalog-service and my IP is 10.0.1.5"). When subscription-service wants to call catalog, it asks Eureka "where is catalog-service?" — Eureka returns the current IP. No hardcoded URLs.

Use case from Disney: When Disney deploys catalog-service on 3 EC2 instances during peak load (e.g., new Marvel movie release), all 3 register with Eureka. Subscription-service automatically load-balances calls across all 3.

Circuit Breaker with Resilience4j

What is Circuit Breaker and why is it NOT the same as Rate Limiter?

Rate Limiter = protects YOUR service from too many incoming requests. "You can only call me 10 times per second."

Circuit Breaker = protects YOU when a service you CALL starts failing. "catalog-service has been failing for 10 seconds, I'll stop calling it and use a fallback instead."

Use case: subscription-service calls catalog-service. If catalog-service goes down, without Circuit Breaker: every subscription request hangs for 30 seconds waiting for timeout → thread pool exhausted → subscription-service also crashes → cascade failure across all microservices. Disney/Netflix call this a "cascading failure" — the #1 production incident cause in microservices.

Circuit Breaker states: CLOSED (normal, calls pass through) → OPEN (circuit tripped, calls fail fast with fallback) → HALF-OPEN (testing if service recovered)

1Add Resilience4j to subscription-service pom.xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
2Configure Circuit Breaker in application.properties
# Circuit Breaker config for catalog-service calls
# slidingWindowSize=10: track last 10 calls
# failureRateThreshold=50: open circuit if 50% of last 10 calls fail
# waitDurationInOpenState=10s: wait 10s before trying again (HALF-OPEN)
# permittedNumberOfCallsInHalfOpenState=3: allow 3 test calls in HALF-OPEN
resilience4j.circuitbreaker.instances.catalog-service.sliding-window-size=10
resilience4j.circuitbreaker.instances.catalog-service.failure-rate-threshold=50
resilience4j.circuitbreaker.instances.catalog-service.wait-duration-in-open-state=10s
resilience4j.circuitbreaker.instances.catalog-service.permitted-number-of-calls-in-half-open-state=3
resilience4j.circuitbreaker.instances.catalog-service.register-health-indicator=true
3Apply Circuit Breaker to SubscriptionService.java
@CircuitBreaker(name = "catalog-service", fallbackMethod = "getPlanFallback")
public OttPlanDTO getPlanFromCatalog(String planId) {
    // Normal call — Feign calls catalog-service
    return catalogServiceClient.getPlanById(planId);
}

// FALLBACK METHOD — called when circuit is OPEN
// Must have same return type + Throwable parameter
public OttPlanDTO getPlanFallback(String planId, Throwable throwable) {
    log.warn("Circuit OPEN for catalog-service. Returning fallback for plan: {}", planId);
    // Return a default/cached response instead of failing completely
    OttPlanDTO fallback = new OttPlanDTO();
    fallback.setPlanId(planId);
    fallback.setPlanName("Plan temporarily unavailable");
    fallback.setIsActive(false);
    return fallback;
}

Kafka Async Messaging — Event-Driven Architecture

Use case — why Kafka in OTT platform

Without Kafka (synchronous): When payment fails → payment-service directly calls notification-service API → if notification-service is slow → payment-service waits → user waits → bad experience.

With Kafka (asynchronous): Payment fails → payment-service publishes event to Kafka topic "payment.failed" → returns 200 to user immediately → notification-service consumes event in background → sends email/SMS → completely decoupled.

Disney's Kafka topics in our OTT platform:

  • payment.failed → notification-service sends failure email
  • payment.success → subscription-service activates the plan
  • subscription.expired → notification-service sends renewal reminder
  • subscription.created → campaign-service checks if promo applies
1Kafka event model — PaymentEvent.java
package com.ott.payment.event;

/**
 * Kafka event published when payment succeeds or fails.
 * This class is shared between producer (payment-service)
 * and consumer (subscription-service, notification-service).
 * In production: put in a shared library or use Avro schema.
 */
public class PaymentEvent {
    private String paymentId;
    private String userId;
    private String planId;
    private String status;  // SUCCESS or FAILED
    private String failureReason;
    private java.time.LocalDateTime eventTime;
    // getters/setters...
}
2Producer — payment-service publishes to Kafka
@Service
public class PaymentEventPublisher {

    private final KafkaTemplate<String, PaymentEvent> kafkaTemplate;

    public PaymentEventPublisher(KafkaTemplate<String, PaymentEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publishPaymentFailed(PaymentEvent event) {
        kafkaTemplate.send("payment.failed", event.getUserId(), event);
        log.info("Published payment.failed event for user: {}", event.getUserId());
    }

    public void publishPaymentSuccess(PaymentEvent event) {
        kafkaTemplate.send("payment.success", event.getUserId(), event);
        log.info("Published payment.success event for user: {}", event.getUserId());
    }
}
3Consumer — notification-service listens to Kafka topics
@Service
public class NotificationKafkaConsumer {

    /**
     * @KafkaListener — Spring automatically calls this method
     * whenever a message arrives on "payment.failed" topic.
     * groupId — consumer group. All instances with same groupId
     * share the load. Only ONE instance processes each message.
     */
    @KafkaListener(topics = "payment.failed", groupId = "notification-group")
    public void handlePaymentFailed(PaymentEvent event) {
        log.info("Received payment.failed for user: {}", event.getUserId());
        // Send email notification
        emailService.sendPaymentFailedEmail(event.getUserId(), event.getFailureReason());
    }

    @KafkaListener(topics = "subscription.expired", groupId = "notification-group")
    public void handleSubscriptionExpired(SubscriptionEvent event) {
        log.info("Received subscription.expired for user: {}", event.getUserId());
        emailService.sendRenewalReminderEmail(event.getUserId(), event.getPlanId());
    }
}

Security — JWT vs OAuth2 — what to use where

Clear explanation — JWT, OAuth2, Spring Security roles

TechnologyWhat it isUse in our project
JWTToken format — stores user identity in a signed string✅ Already implemented — token for API calls
Spring SecurityFramework that secures endpoints, manages users✅ Used in user-service for authentication
OAuth2Protocol for delegated authorizationAdd later for "Sign in with Google"
OIDCOAuth2 + identity (who the user is)Same as above — social login extension

Current architecture (correct for Disney-scale):

  1. User calls user-service: POST /auth/login with username+password
  2. user-service validates credentials against user_db
  3. user-service generates JWT with userId, roles, expiry
  4. Client sends JWT in every request: Authorization: Bearer token
  5. API Gateway validates JWT signature — no DB call needed
  6. Services receive X-Username header — know who is calling

Do you need OAuth2 in each microservice? NO. Only the API Gateway validates tokens. Individual services trust the X-Username header set by Gateway. This is correct — putting auth logic in every service would be a mistake.

Load Balancing — Spring Cloud LoadBalancer + AWS ALB

Two levels of load balancing

Client-side (Spring Cloud LoadBalancer): When subscription-service calls catalog-service and there are 3 instances, Spring Cloud LoadBalancer in subscription-service picks which instance to call (Round Robin / Random). Works with Eureka — gets list of instances from Eureka, distributes calls.

Server-side (AWS ALB): When ESPN website calls your API Gateway and you have 2 gateway instances, AWS ALB distributes incoming traffic across them. External load balancer.

1Add Spring Cloud LoadBalancer to subscription-service
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-loadbalancer</artifactId>
</dependency>

Configure load balancing strategy in application.properties:

# Round Robin (default) — cycles through instances evenly
# Request 1 → Instance 1, Request 2 → Instance 2, Request 3 → Instance 3, repeat
spring.cloud.loadbalancer.ribbon.enabled=false

# For Random strategy — add this config class instead

Once you have Eureka + LoadBalancer, Feign automatically load balances across all catalog-service instances — no code change needed!

2Auto-scaling on AWS — how it works

AWS Auto Scaling: You define rules like "if catalog-service CPU > 70% for 5 minutes → add 2 more instances". AWS automatically launches new EC2 instances, they start catalog-service, register with Eureka, and immediately start receiving traffic.

Centralized Logging — ELK Stack

Problem and solution — use case

Without ELK: User reports "my subscription failed at 3pm". You have 6 services each with their own log file on different servers. You SSH to each server, grep through logs, try to correlate what happened — takes 2 hours.

With ELK: Open Kibana (like Google for your logs), search "userId=12345 timestamp=3pm" — see all logs from all 6 services for that user in one screen, sorted by time. Takes 2 minutes.

What is distributed tracing — use case

Problem: User subscribes → request goes Gateway → subscription-service → catalog-service → payment-service. Where is it slow? Without tracing, you have no way to know which service took how long.

Solution — Zipkin: Each request gets a unique traceId. Every service adds timing info to that trace. Zipkin shows a waterfall chart: Gateway(5ms) → subscription(120ms) → catalog(300ms ← here's the problem!) → payment(50ms). Total: 475ms.

Kubernetes — Container Orchestration

What is Kubernetes — use case

Docker Compose problem: docker-compose runs on ONE machine. If that machine crashes, everything is down. Manual scaling.

Kubernetes (K8s) solution: Runs containers across a CLUSTER of machines. If one machine crashes, K8s automatically moves containers to healthy machines. If load increases, K8s automatically scales to more containers. This is what Disney uses for their streaming platform.

Key K8s concepts:

  • Pod — smallest unit, runs 1+ containers (like one catalog-service instance)
  • Deployment — manages pods, handles scaling, rolling updates
  • Service — stable network endpoint for pods (load balances between pod instances)
  • HPA (Horizontal Pod Autoscaler) — automatically scales pods based on CPU/memory
  • EKS (AWS) — managed Kubernetes on AWS, Disney uses this
1Kubernetes deployment for catalog-service

Master Interview Q&A — all 16 topics

Describe your complete microservices architecture ▼
"I built a Disney-style OTT subscription platform with 6 microservices: user-service for authentication, catalog-service for plan management, subscription-service for user subscriptions, payment-service for billing, notification-service for alerts, and campaign-service for promotions. They communicate through an API Gateway built with Spring Cloud Gateway which handles JWT authentication and rate limiting. For async communication — payment failures and subscription events — I use Kafka. Service discovery is via Eureka. The entire platform is containerized with Docker, monitored with Prometheus and Grafana, and deployable to AWS."
What is the difference between Circuit Breaker and Rate Limiter? ▼
Rate Limiter protects YOUR service from too many incoming requests — it controls how many requests clients can send to you. Circuit Breaker protects YOU when services you call start failing — it detects failures and stops calling the broken service, returning a fallback instead. Rate Limiter = incoming traffic control. Circuit Breaker = outgoing call protection. In our platform, Rate Limiter in API Gateway limits clients to 10 req/sec. Circuit Breaker in subscription-service stops calling catalog-service if it starts timing out, preventing cascade failures.
How does auto-scaling work in your system? ▼
On AWS, I configure an Auto Scaling Group for catalog-service. If CPU usage exceeds 70% for 5 minutes, AWS automatically launches new EC2 instances, installs Docker, pulls the catalog-service image from ECR, and starts it. Each new instance registers with Eureka on startup. Spring Cloud LoadBalancer in subscription-service automatically discovers the new instances and starts distributing Feign calls to them. When load drops, Auto Scaling terminates excess instances. The whole process is automatic and takes about 2 minutes.
What is the role of Kafka in your platform? ▼
Kafka provides asynchronous event-driven communication between services. When a payment fails, payment-service publishes to the payment.failed Kafka topic and immediately returns a response to the user — it doesn't wait for notifications to be sent. notification-service independently consumes from that topic and sends the email. This decoupling means: notification-service going down doesn't affect payment processing, services are independently scalable, and we have a complete audit trail of all events. We also use Kafka for subscription.expired events which trigger renewal reminders.
How would you handle a Disney+ launch day with 10x traffic? ▼
Several layers: API Gateway rate limiter prevents any single user from overwhelming the system. AWS ALB distributes traffic across multiple Gateway instances. Auto Scaling Groups on EC2 automatically scale catalog-service and subscription-service based on CPU metrics. Redis caching in catalog-service means plan data is served from memory, not DB. RDS read replicas handle read-heavy queries. Circuit Breakers prevent cascade failures if any service degrades. For extreme events like a new Star Wars release, we'd pre-scale manually via AWS console, setting minimum instances to 10 in the Auto Scaling Group before the launch time.
Why did you add campaign-service to your project? ▼
I worked on campaign-service professionally at Tavant for the Disney/ESPN project. In that role I maintained the service that managed promotional campaigns — free trial offers, plan upgrade incentives, and subscription renewal promotions. Adding it to my personal project deepened my understanding of the entire system. Campaign-service integrates with subscription-service during checkout to check if any active campaigns apply. It also publishes events to Kafka when campaigns end, triggering notification-service to inform affected users.
