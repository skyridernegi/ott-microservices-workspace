Why Prometheus + Grafana — not just Actuator alone

ToolWhat it doesLimitation alone
Actuator only Shows current snapshot of metrics at /actuator/metrics No history, no graphs, no alerts — you have to manually call the URL
Prometheus Automatically scrapes metrics every 15 seconds, stores history Only raw data — no UI, just text-based queries
Grafana Beautiful dashboards — CPU graphs, request rate charts, error rate alerts Needs a data source like Prometheus — cannot collect metrics itself
All 3 together Complete production monitoring — metrics → stored history → visual dashboards → alerts This is the industry standard stack used at Disney, Netflix, etc.

Step 2 — Configure Prometheus to scrape your services

How Prometheus works: Prometheus is a "pull-based" system. Instead of services pushing metrics to Prometheus, Prometheus periodically calls /actuator/prometheus on each service (called "scraping"). You configure which URLs to scrape in prometheus.yml.
1Create prometheus.yml configuration file
prometheus.yml NEW FILE ott-platform\monitoring\prometheus\prometheus.yml

HOW TO CREATE in Windows: Open Notepad → paste → Save As → navigate to ott-platform\monitoring\prometheus\ → filename: prometheus.yml → Save as type: All Files

# prometheus.yml
# PURPOSE: Tells Prometheus WHERE to scrape metrics FROM and HOW OFTEN
#
# global: default settings for all scrape jobs
# scrape_configs: list of targets to scrape

global:
  scrape_interval: 15s      # scrape every 15 seconds
  evaluation_interval: 15s  # evaluate alerting rules every 15 seconds

scrape_configs:

  # ── Prometheus itself ──────────────────────────────────────────
  # Prometheus monitors itself too — useful to check if it's healthy
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # ── Catalog Service ────────────────────────────────────────────
  # Prometheus calls GET http://catalog-service:8081/actuator/prometheus
  # every 15 seconds and stores the metrics
  # NOTE: In Docker, use container names (catalog-service) not localhost
  - job_name: 'catalog-service'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['catalog-service:8081']
    relabel_configs:
      - source_labels: [__address__]
        target_label: instance
        replacement: 'catalog-service'

  # ── Subscription Service ───────────────────────────────────────
  - job_name: 'subscription-service'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['subscription-service:8082']
    relabel_configs:
      - source_labels: [__address__]
        target_label: instance
        replacement: 'subscription-service'

  # ── API Gateway ────────────────────────────────────────────────
  - job_name: 'api-gateway'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['api-gateway:8080']
    relabel_configs:
      - source_labels: [__address__]
        target_label: instance
        replacement: 'api-gateway'

WHY provisioning: Instead of manually configuring Prometheus data source in Grafana UI every time you restart, provisioning files tell Grafana to auto-configure on startup. Production standard approach.

Create these folders inside your ott-platform folder:

ott-platform\
└── monitoring\
    ├── prometheus\
    │   └── prometheus.yml          ← already created in Step 2
    └── grafana\
        └── provisioning\
            ├── datasources\
            │   └── prometheus.yml  ← create this now (Step 2 below)
            └── dashboards\
                ├── dashboard.yml   ← create this now (Step 3 below)
                └── spring-boot.json ← download link in Step 4 below

Create all folders. In Windows: right-click → New → Folder

2Create Grafana datasource config
prometheus.yml NEW FILE monitoring\grafana\provisioning\datasources\prometheus.yml

WHY: This tells Grafana "connect to Prometheus at http://prometheus:9090 as default data source". Without this you'd have to manually add it in Grafana UI every restart.

apiVersion: 1

datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090    # container name — not localhost
    isDefault: true
    editable: true
    jsonData:
      timeInterval: "15s"          # matches scrape_interval in prometheus.yml
3Create Grafana dashboard provisioning config
dashboard.yml NEW FILE monitoring\grafana\provisioning\dashboards\dashboard.yml

WHY: Tells Grafana to automatically load dashboard JSON files from the dashboards folder on startup.

apiVersion: 1

providers:
  - name: 'OTT Platform Dashboards'
    orgId: 1
    folder: 'OTT Platform'
    type: file
    disableDeletion: false
    editable: true
    updateIntervalSeconds: 30
    options:
      path: /etc/grafana/provisioning/dashboards

Interview Q&A — Monitoring with Prometheus and Grafana

Q: What is the difference between Prometheus and Grafana? ▼
Prometheus is a time-series database that collects and stores metrics. It scrapes /actuator/prometheus endpoints on each service every 15 seconds and stores the data. Grafana is a visualization tool that reads from Prometheus and shows the data as dashboards with graphs, charts, and alerts. Prometheus stores the data, Grafana shows it. Together they form the industry standard monitoring stack for microservices.
Q: What is Micrometer and why is it needed? ▼
Micrometer is a metrics instrumentation library for JVM applications — it's like SLF4J but for metrics. Spring Boot auto-instruments common metrics (JVM memory, HTTP requests, DB connection pools) but stores them in a Spring-internal format. Micrometer acts as a bridge — it converts these Spring metrics into Prometheus format when /actuator/prometheus is called. Without micrometer-registry-prometheus, Prometheus would get a 404 from the Spring Boot service. With it, Prometheus gets properly formatted metrics it can parse and store.
Q: What metrics are most important to monitor in microservices? ▼
The four golden signals of monitoring: Latency (how long requests take — I track http_server_requests_seconds), Traffic (request rate — rate of http_server_requests_seconds_count), Errors (error rate — requests with 5xx status), and Saturation (resource usage — JVM heap memory, CPU, DB connection pool). In our OTT platform I monitor all four — if subscription latency spikes, it could mean catalog-service Feign calls are slow. If error rate increases, it could mean a DB connection pool is exhausted.
Q: How do you distinguish metrics from different microservices in Grafana? ▼
In application.properties I add management.metrics.tags.application=catalog-service. This adds an "application" label to every metric from that service. In Grafana queries I can filter by this label: jvm_memory_used_bytes{application="catalog-service"} shows only catalog memory. Without this tag, all three services would show identical metric names and you couldn't tell which is which.
Q: How does Prometheus scrape metrics — push or pull? ▼
Prometheus uses a pull model. Instead of services pushing metrics to Prometheus, Prometheus periodically calls GET /actuator/prometheus on each service. I configure which services to scrape in prometheus.yml under scrape_configs with the service URL and scrape interval (15 seconds in our setup). This pull model has advantages — Prometheus controls the scrape rate, services don't need to know about Prometheus, and if Prometheus goes down the services are unaffected.
Q: How would you set up alerting in Grafana? ▼
In Grafana, I would configure an alert rule on the error rate panel. For example: if rate(http_server_requests_seconds_count{status=~"5.."}[5m]) > 0.01 (more than 1% error rate for 5 minutes), trigger an alert. I'd configure a notification channel — email or Slack — under Grafana Alerting → Contact Points. In production at Disney-scale, we'd also use Prometheus Alertmanager for more complex alerting logic like suppression, grouping, and routing alerts to different teams based on which service triggered them.
