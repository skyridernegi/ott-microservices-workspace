All Services — Role, Input, Output, DB

user-service
Port: 8083 · DB: user_db
Role: Authentication and user management. The ONLY service that generates JWT tokens.
Input: Registration data, login credentials
Output: JWT token, user profile
APIs: POST /auth/register, POST /auth/login, GET /users/{id}
catalog-service ✅ Built
Port: 8081 · DB: catalog_db
Role: Manages all OTT plans (Disney+, ESPN+, Combo). Read-heavy service.
Input: Plan ID, plan creation data
Output: Plan details, plan list
APIs: GET /catalog/plans, GET /catalog/plans/{id}
subscription-service ✅ Built
Port: 8082 · DB: subscription_db
Role: Manages user subscriptions. Calls catalog via Feign. Calls payment-service. Listens to payment.success Kafka event.
Input: userId, planId, payment result
Output: Subscription details, status
APIs: POST /subscriptions, GET /subscriptions/user/{id}
payment-service 🔥 Build next
Port: 8084 · DB: payment_db
Role: Processes subscription payments. Publishes payment.success or payment.failed to Kafka.
Input: userId, planId, amount, card details
Output: Payment status, receipt
Kafka Produces: payment.success, payment.failed
campaign-service 🔥 Build next
Port: 8085 · DB: campaign_db
Role: Manages promotional campaigns (free trials, discounts). You built this at Disney/Tavant!
Input: Campaign data, promo code, user eligibility
Output: Active campaigns, discount applied
APIs: GET /campaigns/active, POST /campaigns/apply
notification-service 🔥 Build next
Port: 8086 · No DB (stateless)
Role: Pure Kafka consumer. No REST API. Sends email/SMS for payment failures, subscription expiry, renewals.
Kafka Consumes: payment.failed, payment.success, subscription.expired
Output: Email / SMS to user

Key Request Flows — End to End

Flow 1: User Login → Get JWT Token

1
ESPN UI: POST /auth/login {username, password}
↓
2
API Gateway: checks if /auth/login is in PUBLIC_PATHS → YES → forwards without JWT check
↓
3
user-service: validates credentials against user_db → generates JWT with userId, roles, expiry
↓
4
Response: {"token": "eyJhbG...", "userId": 1, "expiresIn": "1 hour"}
↓
5
ESPN UI: stores token in localStorage → sends in Authorization: Bearer header for all future calls

Flow 2: User Subscribes to Disney+ Plan

1
ESPN UI: POST /subscriptions {planId: "PLAN001", paymentDetails: {...}} with JWT token
↓
2
API Gateway: validates JWT → rate limit check → routes to subscription-service:8082
↓
3
subscription-service: validates user exists → calls catalog-service via Feign: GET /catalog/plans/PLAN001
↓
4
subscription-service: checks campaign-service via Feign: GET /campaigns/apply?planId=PLAN001&userId=1 (any promo?)
↓
5
subscription-service: calls payment-service via Feign: POST /payments {userId, planId, amount}
↓
6
payment-service: processes payment → publishes to Kafka: payment.success OR payment.failed
↓
7
notification-service: consumes payment.success from Kafka → sends "Welcome to Disney+!" email
↓
8
subscription-service: listens to payment.success → activates subscription → responds to user: 201 Created

Flow 3: Payment Failure → Notification

1
payment-service: card declined → saves failed payment to payment_db
↓
2
payment-service: publishes PaymentEvent to Kafka topic "payment.failed" → returns immediately (async)
↓
3
notification-service: @KafkaListener receives event → sends "Your payment failed, please retry" email
↓
4
subscription-service: @KafkaListener on payment.failed → keeps subscription in PENDING status

Flow 4: Subscription Expiry Reminder

1
Scheduled job in subscription-service runs daily at midnight: finds subscriptions expiring in 3 days
↓
2
Publishes SubscriptionEvent to Kafka topic "subscription.expiring"
↓
3
notification-service: consumes event → sends "Your subscription expires in 3 days — renew now!" email
↓
4
campaign-service: consumes same event → checks if renewal offer applies → sends promo notification

user_db:
  users: id, username, email, password_hash, full_name, phone, role, is_active, created_at

catalog_db:
  ott_plans: plan_id, plan_name, price, duration, description, is_active, created_at

  users: user_id, username, email, full_name, phone, is_active, created_at
  subscriptions: id, user_id, plan_id, status, start_date, end_date, amount_paid, auto_renewal

  payments: id, user_id, plan_id, amount, status(SUCCESS/FAILED/PENDING),

  campaigns: id, name, description, discount_percent, start_date, end_date,
             eligible_plans, promo_code, max_uses, current_uses, is_active
  campaign_applications: id, campaign_id, user_id, plan_id, discount_applied, applied_at

HLD/LLD Interview Answers

How to walk through HLD in interview (2 minutes)

"Our OTT platform has 6 microservices: user-service for authentication, catalog-service for plan management, subscription-service for user subscriptions, payment-service for billing, campaign-service for promotions, and notification-service for alerts. The ESPN React frontend calls the API Gateway which validates JWT tokens and routes requests. Services communicate synchronously via Feign for data queries and asynchronously via Kafka for events like payment results. Each service has its own MySQL database following the database-per-service pattern. The infrastructure includes Eureka for service discovery, Spring Cloud Config for centralized configuration, Prometheus+Grafana for monitoring, Zipkin for distributed tracing, and ELK for centralized logging. On AWS we use ECS for containers, RDS for databases, MSK for Kafka, and ALB for load balancing."

Why does notification-service have no database?

"Notification-service is a pure event consumer — it only reads from Kafka topics and sends notifications. It has no state of its own to persist. If notification delivery history is needed, we'd add a notifications table, but for our current design, the email/SMS provider (like AWS SES) handles delivery tracking. This keeps the service lightweight and stateless."

Why did you put JWT validation in Gateway and not each service?

"Cross-cutting concerns like authentication should be centralized. If each service validates JWT, you have duplicated auth code across 6 services. Any change to auth logic needs 6 updates. With Gateway as the single enforcement point, each service trusts the X-Username header that Gateway sets — they focus purely on business logic. This is the separation of concerns principle applied at architectural level."
