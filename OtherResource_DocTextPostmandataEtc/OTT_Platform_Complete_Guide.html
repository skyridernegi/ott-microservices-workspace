3 Update pom.xml in Both Services

Your current pom.xml only has Spring Web and DevTools. We need to add MySQL and JPA. Do this for BOTH services.

1Open pom.xml → add these 3 dependencies inside <dependencies>

In STS, expand your project → double-click pom.xml → click the pom.xml tab at the bottom → add these inside the <dependencies> section:

<!-- Spring Data JPA - database operations -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>

<!-- MySQL Driver -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>

<!-- Validation (@NotNull, @NotBlank etc) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
✅ After saving pom.xml, right-click project → Maven → Update Project → OK. STS will download the new dependencies automatically.

9 About JWT Tokens — Do You Need One?

✅ No JWT token needed for Day 1. Both services are completely open — no authentication required. Just call the APIs directly from Postman as shown above.

JWT token validation will be added in Day 2 when we build the API Gateway. The Gateway will sit in front of both services and validate tokens before requests reach your services. Your services themselves stay clean — they don't handle auth at all.

10 Interview Talking Points — What You've Built

When interviewer asks: "Tell me about your microservices project"

Say this:

"I built an OTT subscription platform similar to Disney+/ESPN. It has two microservices — a Catalog Service on port 8081 that manages OTT plans stored in MySQL, and a Subscription Service on port 8082 that handles user subscriptions. When a user subscribes, the Subscription Service calls the Catalog Service via REST to validate the plan and fetch the price. Each service has its own database — catalog_db and subscription_db — following the database-per-service pattern. I used Spring Boot 3.5 with Spring Data JPA, centralized exception handling using @RestControllerAdvice, and DTOs to keep the API contract clean."

Key patterns used (interview terms)

  • Database per service pattern
  • REST-based synchronous communication
  • Centralized exception handling (@RestControllerAdvice)
  • DTO pattern (Request/Response separation)
  • Repository pattern (Spring Data JPA)
  • Soft delete (deactivate, not hard delete)
  • Separation of concerns (Controller → Service → Repository)

Common interview questions

  • What is the difference between @RestController and @Controller?
  • Why use DTO instead of Entity directly?
  • What is @RestControllerAdvice?
  • How do microservices communicate?
  • Why give each service its own database?
  • What is soft delete and why use it?
  • What is @Transactional and when to use it?
